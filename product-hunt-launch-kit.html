Campaign summary

Thinkpool should launch as a pair-programming workspace around a connected coding agent, not as a broad AI workspace or multi-agent command center.

The five-second idea is simple: two people work out the direction in one live code room, then use @pool to hand the on-record room context to the agent. A // aside stays between the two humans. The agent work remains visible in the room.

Primary audience: technical cofounders and trusted product-engineering pairs already using terminal coding agents. The triggering situation is, “Can you jump in and help steer this?” during ambiguous, risky, or product-sensitive work.

Main proof: the pair discussion, explicit @pool handoff, connected-agent activity, and visible review. Supporting proof: bounded visible lanes. The campaign does not lead with Cascade, Flow, Design, research, mobile, model breadth, or agent count.

Primary CTA: Start a room with the person who should help make the next coding decision.

Full description

Coding agents can make a lot of changes before two people agree on the right one.

Thinkpool gives a human pair one live code room around a connected coding agent. Discuss the work together, use // when an aside should stay between the two humans, then use @pool to hand the on-record room context to the agent. The handoff, agent activity, and review remain visible in the room.

One person runs the small thinkpool-pair bridge on the machine with the repository and coding agent. Both people can watch and steer through the browser while that bridge stays connected. Claude Code and Codex are the public supported runtimes today. The bridge host brings the provider setup, and provider charges are separate.

Thinkpool is not another coding model, an always-on cloud agent, or a replacement for the editor and agent you already prefer. If the bridge disconnects, the room remains a readable record, but it does not switch to a hidden fallback AI.

Start a room with the person who should help make the next coding decision.

FAQ

What is Thinkpool?

Thinkpool is a shared live code room for two people and a coding agent running through one connected machine. It gives the pair one place to discuss the work, make the handoff explicit, and follow the result.

Who is it for?

Technical cofounders, product-engineering pairs, and trusted collaborators who should share a live coding decision. It is most useful when the second person needs to understand and redirect the work, not merely watch a screen.

Is Thinkpool another coding agent?

No. It is the room around the coding agent you already use. Claude Code and Codex are the public supported runtimes today.

How is this different from several agent terminals?

Several terminals provide parallel execution. Thinkpool focuses on the pair’s decision path around that execution: on-record discussion, an optional human-only aside, an explicit @pool handoff, and visible follow-on work in the same room. For independent solo tasks, several terminals may be the better tool.

Why not Slack plus a terminal or screen share?

That setup works for many teams. Thinkpool is useful when reconstructing the decision in another tool becomes the problem. The discussion, handoff, agent activity, and review stay together. It does not claim Slack or screen sharing cannot support collaboration.

How does it compare with Warp, Agor, Replit, Zed, Factory, or Codex?

Those products cover meaningful parts of collaborative development and multi-agent work. Thinkpool does not claim to be the only shared agent surface. Its focus is narrower: the human pair’s decision and the explicit handoff to a connected coding agent.

Do both collaborators need desktop access?

No desktop app is required for the collaborator. Both people use the browser room, including from a mobile browser, while one participant keeps the bridge connected on the machine hosting the repository and coding agent. The phone does not run the bridge or the code.

What happens if the bridge disconnects?

The room remains a readable transcript. Thinkpool does not switch to a hidden hosted model, and new agent work waits for the bridge host to reconnect.

What does @pool do?

It packages the on-record room conversation plus a bounded digest of visible sibling lanes and sends that context to the same bridge-hosted connected agent. It is not a separate synthesis model and does not broadcast a command to every lane.

Is // a secure private channel?

No. It is a human-only routing boundary in the room. The aside stays out of the default agent and @pool context, but it should not be described as encrypted or as a confidentiality guarantee.

Whose provider key and bill does it use?

The bridge host configures the provider setup. The key is sealed to that bridge, the provider call happens from the host machine, and provider charges are separate from Thinkpool. Claude lanes accept Anthropic-compatible endpoints directly. Non-Anthropic models require a translating gateway such as LiteLLM. Codex uses its Codex/OpenAI login.

Is it safe for a private repository?

Do not answer this with a blanket assurance. Repository files remain on the bridge host, but room transcripts, attachments, activity records, and presence are handled under Thinkpool’s Privacy Policy. Voice dictation sends audio to Groq for transcription. Teams should review the current policy and access model against their requirements.

What is available today?

Available: two-person rooms, the connected bridge, //, @pool, visible bounded lanes, Claude Code and Codex support, provider-direct operation, browser access, and rendered Design/mockup cards.

Available with conditions: mobile-browser participation while the host bridge remains connected; compatible providers and gateways; bounded Ensemble/Cascade delegation.

Not a launch promise: a bridge-down fallback AI, native mobile app, arbitrary web-app editing, unlimited agents, bundled model usage, or every detected CLI working identically.

Skeptical-question response kit

These are answer structures, not copy-paste comments. A named maker should respond to the person’s exact context.

“This sounds like extra ceremony.”

It is extra ceremony when one person already knows what to build. We are targeting the cases where two people genuinely own the decision and a premature prompt is expensive. If that is not your workflow, your current terminal setup is probably better.

“Why not just open more agents?”

More agents help with throughput. Thinkpool is aimed at shared understanding: can the second person see the same decision, challenge it, and redirect the resulting work without becoming a courier between windows?

“Can my cofounder join from a phone?”

Yes, through the mobile browser while the bridge host stays connected on the machine with the repository and agent. The phone participates in the room; it does not run the bridge or execute the code.

“Does it need my API key?”

The bridge host configures the provider setup and pays that provider directly. The key is sealed to the bridge. Thinkpool does not bundle model usage into the hosted-room price.

“What happens when it breaks?”

If the bridge disconnects, the room stays readable but new agent work stops until it reconnects. We prefer showing that condition to silently moving the work to another model.

Questions that must be answered personally, not from a template:

  • Which runtime and provider configuration did you personally test in the launch build?
  • What exact permissions can each participant exercise?
  • How are invites, removal, room retention, and deletion handled?
  • What happened in a specific failure, billing, privacy, or data-loss report?
  • What remains incomplete, and what must a pair still do manually?
